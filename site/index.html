oatcake.css: vanilla typography stylesheet

A drop-in CSS file that adds nice typography to a web page. Aims to look good while not attracting attention to itself or getting in the way of reading or writing. Works great on mobile.

This page is styled using oatcake.css, my basic typography stylesheet. To add it to your own page just add this to your HTML file's <head>:

<link rel="stylesheet" href="https://www.seanh.cc/oatcake/oatcake.css">

There are hundreds of these but I couldn't find a really good one that I liked, so Oatcake is my contribution to the genre. It has all the typographical features that I wanted, including vertical rhythm, figures with captions, blockquotes with citations, note boxes and disclosure boxes, badges and pills, lead paragraphs and subtitles, and more.

You can throw a wide variety of different content and combinations of HTML elements at Oatcake and everything will look good: it handles all the edge-cases that I kept finding broken in other themes. Things like multiple levels of headings, nested lists and blockquotes, definition lists, headings that wrap onto multiple lines, headings inside blockquotes, inline elements within headings, inline images, embedded videos and iframes, etc.

I recommend setting a line width of about 30-50em using the CSS max-width property on a container element like a <main>, an <article> or just a <div> that contains your content. Here's a complete starter page:

<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Page title</title>
    <link rel="stylesheet" href="https://www.seanh.cc/oatcake/oatcake.css">
  </head>
  <body>
    <main style="max-width:50em; margin:0 auto 0 auto; padding:0 .5em 0 .5em;">
      <h1>Page title</h1>
      <p>Page content goes here...</p>
    </main>
  </body>
</html>

The rest of this page demonstrates and discusses Oatcake's features and design choices. Plus, I needed a comprehensive test page.

The table of contents is generated by tocbot, which you can easily add to any page.

If you hover the mouse over any heading on this page you'll notice that they all have clickable #anchor links attached to them for deep linking. For example here's a deep link to the the "Lists" section. The anchor links are created by AnchorJS, just add this to the bottom of your page's <body>:

Vertical rhythm

Oatcake sets a nice, comfortable base font-size and line-height and aligns all the other sizes, margins, paddings and borders with that line-height for a consistent vertical rhythm, like writing on a sheet of lined paper. Headings

HTML provides six levels of headings, more than most documents need. Trying to make all six levels visually distinct results in distractions like overlarge font sizes, overheavy font weights, ALL CAPS, underlines, centered text, too many different fonts, etc.

Most pages should use no more than three levels of headings, preferably two. More levels make it difficult and confusing for readers to orient themselves within the structure of the text. So Oatcake provides three levels of clearly distinct headings, using the smallest necessary increments in font size and space above and below to emphasise and distinguish headings without the design attracting too much attention.

A common mistake is to vertically align headings closer to the paragraph above than to the one below, or evenly mid-way between them. Oatcake gives headings a little more vertical space above them than below so that they're visually related more with the text below (that belongs to the heading) than to the unrelated text above. <h1> and <h2> headings are the same size (other than a bit more vertical space above an <h1>). The assumption is that pages will start with a single <h1> whose position at the top of the page makes it clear that it's the top-level page title, and then <h2>'s will be used for any mid-document headings.

If you need sub-headings and sub-sub-headings use <h3>'s and <h4>'s.

When to use figures

According to the HTML spec on <figure> a figure is supposed to be a unit of content that:

…is self-contained, that is typically referenced as a single unit from the main flow of the document, and that can be moved away from the main flow of the document without affecting the document's meaning.

If the location of an image or other content within the document is important to the meaning of the document then you should just use a standard <img>, <pre><code>, or other content instead of a figure.

Representing user input: <kbd> and <samp>

Oatcake supports user input with the <kbd> element and sample output with the <samp> element.

To represent user input from a keyboard or any other text entry device wrap it in <kbd> tags: help mycommand

You can use nested <kbd>'s to represent multiple keystrokes that form a single input: Ctrl + c. Although you might prefer to write it without nesting like this: Ctrl + c.

Up to two levels of nesting are supported: Ctrl + b Shift + t.

To represent sample or quoted output from a computer program, wrap it in <samp> tags: Keyboard not found. Press F1 to continue

Tip: use non-breaking spaces between <kbd>'s

Oatcake prevents browsers from wrapping a <kbd> (nested or single) over multiple lines even if the <kbd> contains multiple words separated by spaces, but if you're using a chain of separate <kbd>'s (as in Ctrl + c) you must use non-breaking spaces (&nbsp;) instead of normal spaces around the + if you want to prevent line breaks from being inserted in-between the <kbd>'s. Wrapping the whole sequence of <kbd>'s in a <span style="white-space: nowrap;"> should also work.

You can also use either technique if you don't want a <code>, <samp>, <mark>, etc to be wrapped (Oatcake doesn't prevent wrapping within these elements by default because often you do what them to wrap, especially if they contain long strings that might otherwise overflow the container).

Preventing really long words from overflowing

Long unbroken words or other strings (such as URLs) that are too long to fit on one line (even with a whole line to themselves) can overflow their containers, especially on mobile, causing unwanted horizontal scrollbars or other layout breakage. Oatcake fixes this by wrapping long words over multiple lines if necessary to prevent overflow.

For example this really long Sanskrit word will get wrapped: िरन्तरान्धकारित-दिगन्तर-कन्दलदमन्द-सुधारस-बिन्दु-सान्द्रतर-घनाघन-वृन्द-सन्देहकर-स्यन्दमान-मकरन्द-बिन्दु-बन्धुरतर-माकन्द-तरु-कुल-तल्प-कल्प-मृदुल-सिकता-जाल-जटिल-मूल-तल-मरुवक-मिलदलघु-लघु-लय-कलित-रमणीय-पानीय-शालिका-बालिका-करार-विन्द-गलन्तिका-गलदेला-लवङ्ग-पाटल-घनसार-कस्तूरिकातिसौरभ-मेदुर-लघुतर-मधुर-शीतलतर-सलिलधारा-निराकरिष्णु-तदीय-विमल-विलोचन-मयूख-रेखापसारित-पिपासायास-पथिक-लोकान्.

So will this really long URL in a <code> tag: https://github.com/seanh/oatcake/commit/8675ffcb168b1d76d708c84a2821c20a3f99e86d.

Oatcake doesn't wrap long words inside preformatted text (<pre>) blocks -- you don't want your code being wrapped for you, these use a horizontal scrollbar instead: 

url = 'https://github.com/seanh/oatcake/commit/8675ffcb168b1d76d708c84a2821c20a3f99e86d'

Tip: avoid long text in a <kbd>, <badge> or <pill>

Oatcake's CSS prevents line-wrapping within a <kbd>, <badge> or <pill> (even if there are spaces) so don't make these too long or they'll overflow their container and break the layout.
