Oatcake: plain web typography

A drop-in stylesheet that makes any HTML page's text attractive and readable, no CSS classes needed. Looks good without drawing attention to the design, lots of typographic features and details, everything works great on mobile.

This page is a demo of Oatcake, my universal typography stylesheet. To add Oatcake to your own page just add this link to your HTML file's <head>:

<link rel="stylesheet" href="https://www.seanh.cc/oatcake/oatcake.css">

You don't need any CSS classes or any particular HTML structure: Oatcake styles the standard HTML elements directly. You can use it to style HTML straight from Markdown.

For more documentation and examples see Oatcake's README on GitHub.

There are loads of these typography stylesheets out there but I couldn't find one that was a great fit for my tastes and the type of content that I write, or that handled all the minutiae and edge-cases that I care about without issues.

Oatcake has lots of typographic features including a system font stack, vertical rhythm, figures and captions, blockquotes, note boxes, disclosure widgets, badges and pills, multiple levels of headings, subtitles and lead paragraphs, <kbd> and <samp> styles, and more.

My pages tend to have a wide variety of content and different combinations of HTML elements. Oatcake carefully handles many situations that break most themes: things like nested lists, definition lists, multi-paragraph blockquotes, multi-line headings, headings with code and links in them, headings inside blockquotes, wrapping long words and wide code blocks, images and other media, etc.

The rest of this page demonstrates and discusses Oatcake's features and design decisions. Plus I needed a comprehensive test page. You may even learn something about HTML along the way!

The table of contents is generated by Tocbot. If you hover the mouse over headings on this page they have clickable #anchor links, those are generated by AnchorJS.

System font stack

Oatcake uses a system font stack based on the system-ui font-family: it chooses the default interface font from the user's OS. This is fast (no fonts to download), readable (modern OSes ship high-quality fonts adapted to today's screens), and feels familiar and native to the user's system (no distracting idiosyncratic font choices).

Code blocks use a system monospaced font stack based on the ui-monospace font-family which selects the OS's default monospaced font.

Vertical rhythm

Oatcake sets a comfortable base font-size and line-height and aligns all the other sizes, margins, paddings and borders with that vertical rhythm, like writing on a sheet of lined paper. This makes everything harmonious and pleasant to read. You can turn on guide lines to see what I mean:

Inline HTML elements you can use

Here's a pocket guide to some of the more interesting inline elements you can use with links to the HTML standard. These all work well with Oatcake:

  • The <em> element is for stress emphasis. <em> is what *single asterisks* or _single underscores_ do in Markdown.

    Not to be confused with <i> which is for a span of text in an alternate voice or mood, or otherwise offset from the normal prose in a manner indicating a different quality of text, such as a taxonomic designation, a technical term, an idiomatic phrase from another language, transliteration, a thought, or a ship name in Western texts. A dream sequence is another example of <i> given in the HTML standard.

    Both are styled in italics:

    This is emphasised text with <em>.
    This is text in an alternative voice or mood with <i>.

    If some text is in an <i> because it's in a different language you can use the lang attribute to give the language:

    <p>There is a certain <i lang="fr">je ne sais quoi</i> in the air.</p>

  • The <strong> element is for strong importance, seriousness or urgency. <strong> is what **double asterisks** or __double underscores__ do in Markdown.

    Not to be confused with <b> which represents a span of text to which attention is being drawn for utilitarian purposes without conveying any extra importance and with no implication of an alternate voice or mood, such as key words in a document abstract, product names in a review, actionable words in interactive text-driven software, or an article lede.

    Both are styled in bold:

    This is important text with <strong>.
    This is text to which attention is being drawn for utilitarian purposes with<b>.

    You should use <b> only as a last resort when no other element is more appropriate. Use <h1> to <h6> for headings, <em> for stress emphasis, <strong> for importance, and <mark> for highlighted or marked text.

  • The <small> element is for side comments and small print like disclaimers, caveats, legal restrictions or copyrights, attribution, etc. It's styled in small text: this is some small text created with the <small> element.

    <small> is only for short runs of text, use <aside> for larger sidebars, pull quotes, groups of <nav> elements, etc. Use <hgroup> for subheadings.

  • The <q> element is for inline quotations: Whatever you lose, you've won. Whatever you win, you've lost.

    You don't have to type the quotation marks (“…”) yourself, just use <q>…</q> and the browser will add the quotation marks for you.

    You can nest <q>'s to represent nested quotations: The man said Things that are impossible just take longer. I disagreed with him.

    <q> is only for quoting another source. If quotation marks are being used for another reason (for example to mark up sarcasm) you should just type the quotation marks yourself. The HTML standard gives this example of typing quotation marks directly when <q> would be inappropriate:

    The word "ineffable" could have been used to describe the disaster resulting from the campaign's mismanagement.

    For block quotations see the section on blockquotes below.

    You can add a cite attribute to a <q> to give the URL of the source being quoted for scripts to read, and you can use a <cite> element before or after a <q> to mark up a citation for the quote. See the section on blockquotes below for more about cite and <cite>.

  • The <abbr> element is for abbreviations and acronyms. If you provide the expansion of the abbreviation as the value of a title attribute on an <abbr> then in some browsers the abbreviation will get a dotted underline and the expansion will be shown in a tooltip on mouseover. For example this: <abbr title="Hypertext Markup Language">HTML</abbr> renders like this: HTML.

  • The <code> element is for inline code spans: fragments of code or strings, filenames, computer programs, etc. This is what `backticks` do in Markdown. See code blocks below for multi-line code blocks.

  • Instead of <code>, you can also use <var> for programming, mathematical and prose variables, constants, symbols, quantities, parameters and placeholders. Here's an example from the HTML standard:

    If there are n pipes leading to the ice cream factory then I expect at least n flavors of ice cream to be available for purchase!

  • The <samp> element represents output from a program and <kbd> represents user input. See the section on <kbd> and <samp> below for more.

  • The <sub> and <sup> elements are for subscript and superscript.

    Superscripts can be used to create footnotes.[1]

    The HTML standard gives some interesting examples of <sub> and <sup> used with <var> for variables or mathematical expressions like xi or c2.

  • The <u> element gives a span of text an unarticulated, though explicitly rendered, non-textual annotation, such as labeling the text as being a proper name in Chinese text (a Chinese proper name mark), or labeling the text as being misspelt. It's styled with an underline: this is text annotated with <u>.

    Like <b>, <u> is another last-resort element: use <em> for stress emphasis, <b> or <mark> to mark key words or phrases, <cite> to mark book titles, <i> for technical terms, taxonomic designations, transliterations, or thoughts. <u>'s are also discouraged where users might confuse them with hyperlinks (<a>) which are conventionally rendered with an underline.

  • The <mark> element is for denoting the relevance of a span of text to the current context (as opposed to denoting importance, which is done with <strong>, or emphasis, which is done with <em>). <mark>'s are styled with a background highlight: this is some text marked as relevant using <mark>.

    More specifically, a <mark> represents: a run of text in one document marked or highlighted for reference purposes, due to its relevance in another context. For example you can use a <mark> in a quotation to highlight text that wasn't highlighted in the original quote but is being highlighted now to bring attention to part of the quote. Other examples given in the HTML standard are highlighting search string matches and highlighting a bug in a code block (or more generally highlighting part of a code block that is referred to from elsewhere in the text).

  • The <del> element is for text that has been "removed" from the document like crossing-off the completed items on a todo list, whereas <s> is for text that's no longer accurate or relevant. Both are styled with a strike-through:

    This is text that's been struck through with <del> to remove it from the document.
    This is text that's been struck through with <s> to mark it as no longer accurate or relevant.

    The cite attribute can be added to a <del> to give the URL of a document that explains the change, and a datetime attribute can be used to give the date and time of the change.

  • The <ins> element is for text that has been added to the document. It's styled with an underline the same as <u>: this is text that has been marked as added to the document with <ins>.

    As with <del>, you can add a cite attribute to an <ins> to give the URL of a document that explains the change, and a datetime attribute to give the date and time of the change.

    You can represent items added to or removed from a list by wrapping the contents of <li>'s in <ins>'s or <del>'s:

Headings

Headings are one of the hardest things to get right. HTML provides six levels of headings, far more than most documents need. Themes trying to make all six levels visually distinct results in distractions like overlarge font sizes, excessively heavy weights, ALL CAPS, underlines, centered text, too many font faces, too much indentation, etc. Most themes make <h1>'s too big for my taste (at least with the types of titles I tend to write).

Most pages should use no more than three levels of headings anyway, preferably two. More levels are confusing and make it difficult for readers to use the headings to orient themselves within the document.

Oatcake focuses on providing three visually distinct levels of headings, using the smallest necessary increments in font size and space above and below to emphasise and distinguish headings without the design attracting too much attention.

A common mistake is to vertically align headings closer to the paragraph above than to the one below, or evenly mid-way between them. Oatcake gives headings a little more vertical space above than below so they're visually related more with the text below (that belongs to the heading) than to the unrelated text above.

<h1>'s and <h2>'s look the same (other than a bit more vertical space above an <h1>). The intention is that pages will start with a single <h1> whose position at the top of the page makes it clear that it's the top-level title, and then <h2>'s will be used for any mid-document headings.

If you need sub-headings and sub-sub-headings use <h3>'s and <h4>'s.

You can also use <h5>'s and <h6>'s but they look the same as <h4>'s.

Definition lists

The <dl> (definition list) element represents:

…an association list consisting of zero or more name-value groups (a description list)

This could be a list of …terms and definitions, metadata topics and values, questions and answers, or any other groups of name-value data.

Within a <dl> you use pairs of <dt>'s for the names ("terms") and <dd>'s for the values ("descriptions" or "definitions"). You can have multiple <dt>'s in a row if an item has more than one name and/or multiple <dd>'s in a row if an item has more than one value. Here's some examples from the HTML standard:

<dl>
 <dt> Authors
 <dd> John
 <dd> Luke
 <dt> Editor
 <dd> Frank
</dl>
<dl>
 <dt lang="en-US"><dfn>color</dfn></dt>
 <dt lang="en-GB"><dfn>colour</dfn></dt>
 <dd>A sensation which (in humans) derives from the ability of
 the fine structure of the eye to distinguish three differently
 filtered analyses of a view.</dd>
</dl>

Notice the use of the <dfn> element to mark up the terms when a definition list is being used to define terms. Oatcake doesn't style <dfn>'s but they have semantic meaning.

Blockquotes

You can use <q> for inline quotes. For multi-line or block quotes use <blockquote>, which looks like this:

Ut fermentum turpis quis ligula sagittis, a fringilla orci ultrices. Nam fringilla erat ac bibendum tristique. Morbi nec maximus augue. Donec in justo semper, convallis ante et, laoreet nibh.

You can optionally wrap the contents of a <blockquote> in a <p>, it looks the same:

Maecenas eu purus in nibh pharetra gravida. Nullam consequat fringilla velit, nec maximus ex luctus quis. Maecenas tempor sollicitudin nisl ac consequat. Nullam ut iaculis tortor.

Blockquote usage tips

You shouldn't put quotation marks (“…”) around the text inside a blockquote: putting the text in a <blockquote> is enough to show that it's a quotation, and the quotation marks aren't part of the quote itself. You don't put quotation marks inside an inline <q> either, just wrap the text in a <q> and the browser will add the quotation marks for you.

In English it's traditional to use square brackets if abbreviating or adding context to a block or inline quote, as in this example from the HTML standard (which also handily demonstrates that one-liner <blockquote>'s also work):

[Jane] then said she liked [...] fish.

You can use <mark>'s inside quotes to highlight something that wasn't highlighted in the original quote:

Contrary to popular belief, Lorem Ipsum is not simply random text. It has roots in a piece of classical Latin literature from 45 BC, making it over 2000 years old. Richard McClintock, a Latin professor at Hampden-Sydney College in Virginia, looked up one of the more obscure Latin words, consectetur, from a Lorem Ipsum passage, and going through the cites of the word in classical literature, discovered the undoubtable source. Lorem Ipsum comes from sections 1.10.32 and 1.10.33 of "de Finibus Bonorum et Malorum" (The Extremes of Good and Evil) by Cicero, written in 45 BC.
Attributions and citations

A <q> or <blockquote> represents text that is quoted from another source. You can add a cite attribute whose value is a URL for the source:

<blockquote cite="https://quotes.example.org/s/sonnet130.html">

cite attributes aren't visible in the rendered HTML, they're meant to be read by scripts.

Any attribution for the quotation is supposed to go before or after the <q> or <blockquote>, not inside it, because the attribution isn't part of the quote itself. For example the attribution could go in a following <p>, as in this example from the HTML standard:

I contend that we are both atheists. I just believe in one fewer god than you do. When you understand why you dismiss all the other possible gods, you will understand why I dismiss yours.

â€” Stephen Roberts

If the attribution contains the title of a work you can wrap it in a <cite> element, as in this example also from the HTML standard:

<p>His next piece was the aptly named <cite>Sonnet 130</cite>:</p>
<blockquote>
  <p>My mistress' eyes are nothing like the sun,<br>
  Coral is far more red, than her lips red,<br>
  ...

<cite> elements aren't styled any differently from normal text, they're semantic markup only.

Only the title of the work should go inside the <cite>, not an entire phrase or sentence.

You shouldn't wrap an author's name in a <cite> because it's not the title of a work.

<cite> can be used whenever you're mentioning the title of a work and want to mark it up semantically, even if you're not quoting from the work.

You can put a <blockquote> in a <figure> and the attribution in the <figure>'s <figcaption>, see the section on figures below for an example.

Code blocks can created with <pre><code> tags. This is what indenting a block of text by four spaces or one tab does in Markdown (or "fenced code blocks" with ``` in many versions of Markdown). Oatcake gives code blocks a monospaced font:

#include <stdio.h>

int main()
{
    printf("Hello World!\n");
}

Oatcake also supports <pre><samp> blocks. These have a different semantic meaning in HTML: they represent a block of computer output. Oatcake styles <pre><samp> the same as <pre><code>:

$ gcc -g -o helloworld -ansi helloworld.c
$ gdb helloworld
(gdb) run
Hello World
(gdb) quit

You can use inline elements in preformatted, code and sample output blocks: bold, italics, underline, strikethrough, small text. Here's one with a bunch:

    /* Here's a code comment in small text. */
    /* Here's an inline quote in a code block. */
    printf("Hello World!\n");

A useful application of inline elements in code blocks, from the HTML standard, is using <mark> to highlight part of the code, perhaps a part that is referred to from elsewhere in the text. For example here's a code block with a bug highlighted:

def whats_on_the_telly(penguin=[]):
    penguin.append("property of the zoo")

A common bug in themes is for long lines in code blocks to stretch out the width of the entire page which looks weird and breaks the layout on mobile screens. Oatcake fixes this by giving wide code blocks a horizontal scrollbar:

def test_validate_url_returns_an_http_url_unmodified():

Input and output with <kbd> and <samp>

Oatcake supports user input with the <kbd> element and sample output with the <samp> element.

To represent user input from a keyboard or any other text entry device wrap it in <kbd> tags: help mycommand

You can use nested <kbd>'s to represent multiple keystrokes that form a single input: Ctrl + c. Although you might prefer to write it without nesting like this: Ctrl + c.

Up to two levels of nesting are supported: Ctrl + b Shift + t.

To represent sample or quoted output from a computer program, wrap it in <samp> tags: Keyboard not found. Press F1 to continue

Tip: use non-breaking spaces between <kbd>'s

Oatcake prevents browsers from wrapping a <kbd> (nested or single) over multiple lines even if the <kbd> contains multiple words separated by spaces, but if you're using a chain of separate <kbd>'s (as in Ctrl + c) you must use non-breaking spaces (&nbsp;) instead of normal spaces around the + if you want to prevent line breaks from being inserted in-between the <kbd>'s. Wrapping the whole sequence of <kbd>'s in a <span style="white-space: nowrap;"> should also work.

You can also use either technique if you don't want a <code>, <samp>, <mark>, etc to be wrapped (Oatcake doesn't prevent wrapping within these elements by default because often you do what them to wrap, especially if they contain long strings that might otherwise overflow the container).

When to use figures

According to the HTML spec on <figure> a figure is supposed to be a unit of content that:

…is self-contained, that is typically referenced as a single unit from the main flow of the document, and that can be moved away from the main flow of the document without affecting the document's meaning.

If the location of an image or other content within the document is important to the meaning of the document then you should just use a standard <img>, <pre><code>, or other content instead of a figure.
